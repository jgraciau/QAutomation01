# Guía de Contribución - QAutomation01\n\nGracias por tu interés en contribuir a QAutomation01. Este documento proporciona directrices para contribuir al proyecto.\n\n## 📋 Código de Conducta\n\nEste proyecto se adhiere a un código de conducta profesional. Esperamos que todos los contribuidores:\n\n- Respeten la diversidad de opiniones\n- Creen un ambiente inclusivo y acogedor\n- Se comuniquen de manera constructiva\n- Enfoquen en el mejoramiento del código\n\n## 🚀 Comenzar\n\n### 1. Fork y Clonar\n\n```bash\n# Fork el repositorio en GitHub\n# Luego clona tu fork\ngit clone https://github.com/tu-usuario/QAutomation01.git\ncd QAutomation01\n\n# Agrega el repositorio original como upstream\ngit remote add upstream https://github.com/original/QAutomation01.git\n```\n\n### 2. Instalar Dependencias\n\n```bash\nnpm install\nnpx playwright install\n```\n\n### 3. Crear Rama Feature\n\n```bash\n# Actualiza main\ngit fetch upstream\ngit checkout main\ngit merge upstream/main\n\n# Crea tu rama feature\ngit checkout -b feature/descripcion-corta\n# o\ngit checkout -b fix/descripcion-corta\n# o\ngit checkout -b docs/descripcion-corta\n```\n\n## 🎯 Proceso de Contribución\n\n### Tipos de Contribuciones\n\n#### 1. Nuevos Tests\n\n- Crear suite en `tests/` organizada por funcionalidad\n- Usar Page Objects para todas las interacciones\n- Documentar con comentarios descriptivos\n- Asegurar que tests sean independientes\n\n**Ejemplo:**\n\n```typescript\n// tests/opencart/new-feature.spec.ts\nimport { test, expect } from '@playwright/test';\nimport { HomePage } from '../../src/pages/HomePage';\n\ntest.describe('Feature X', () => {\n  let homePage: HomePage;\n\n  test.beforeEach(async ({ page }) => {\n    homePage = new HomePage(page);\n    await page.goto('common/home');\n  });\n\n  test('describes what the test does', async ({ page }) => {\n    // Test implementation\n  });\n});\n```\n\n#### 2. Nuevos Page Objects\n\n- Crear clase en `src/pages/`\n- Encapsular todos los selectores\n- Documentar métodos con JSDoc\n- Incluir validaciones y error handling\n\n**Ejemplo:**\n\n```typescript\n// src/pages/NewPage.ts\nimport { Page, Locator } from '@playwright/test';\n\n/**\n * Page Object para nueva página\n * @class NewPage\n */\nexport class NewPage {\n  readonly page: Page;\n  readonly element: Locator;\n\n  constructor(page: Page) {\n    this.page = page;\n    this.element = page.locator('#element-id');\n  }\n\n  /**\n   * Descripción de la acción\n   * @throws Error si no se puede realizar\n   */\n  async actionName(): Promise<void> {\n    await this.element.click();\n  }\n}\n```\n\n#### 3. Utilidades\n\n- Agregar a `src/utils/TestHelpers.ts`\n- Documentar parámetros y retorno\n- Incluir ejemplos de uso\n- Reutilizable en múltiples tests\n\n#### 4. Documentación\n\n- Actualizar README.md si cambios relevantes\n- Agregar ejemplos de código\n- Explicar decisiones arquitectónicas\n- Mantener en español\n\n## ✅ Antes de Hacer Commit\n\n### 1. Código\n\n```bash\n# Verificar que los tests pasan\nnpm run test\n\n# Ejecutar en modo visual para verificar\nnpm run test:headed\n\n# Ejecutar UI para debugging visual\nnpm run test:ui\n```\n\n### 2. Lint y Format (Opcional pero recomendado)\n\nSi tienes ESLint configurado:\n\n```bash\nnpx eslint src/ tests/ --fix\n```\n\n### 3. Nombres de Variables\n\n- Usar camelCase para variables y funciones\n- Usar PascalCase para clases\n- Nombres descriptivos y en inglés para código\n- Comentarios en español\n\n```typescript\n// ✅ BIEN\nconst userFirstName = 'John';\nclass CheckoutPage {}\n\n// ❌ MAL\nconst ufn = 'John';\nclass checkout_page {}\n```\n\n### 4. Documentación\n\n- Cada función/clase debe tener comentario JSDoc\n- Incluir `@param`, `@returns`, `@throws` si aplica\n- Agregar ejemplos de uso con `@example`\n\n```typescript\n/**\n * Selecciona una opción del dropdown\n * @param {Locator} selectLocator - Elemento select\n * @param {string} label - Etiqueta de opción\n * @returns {Promise<void>}\n * @throws {Error} Si la opción no existe\n * @example\n * await TestHelpers.selectOptionByLabel(page.locator('#country'), 'USA');\n */\nstatic async selectOptionByLabel(selectLocator: Locator, label: string): Promise<void> {\n  // Implementación\n}\n```\n\n## 📝 Commits\n\n### Formato de Mensajes\n\n```\n<tipo>(<escopo>): <descripción corta>\n\n<descripción detallada opcional>\n\n<referencias a issues opcionales>\n```\n\n### Tipos de Commits\n\n```\nfeat:     Nueva funcionalidad o test\nfix:      Correción de bug\ndocs:     Cambios en documentación\ntest:     Agregar o actualizar tests\nrefactor: Restructuración sin cambios funcionales\nstyle:    Cambios de estilo sin afectar código\nchore:    Cambios en configuración, build, etc\n```\n\n### Ejemplos\n\n```bash\ngit commit -m \"feat(checkout): add guest checkout test\"\n\ngit commit -m \"fix(page-object): correct selector for billing form\n\nFixed issue where billing form selector was too generic.\nNow using more specific data-testid selector.\n\nCloses #123\"\n\ngit commit -m \"docs(readme): add troubleshooting section\"\n\ngit commit -m \"test(opencart): improve timeout handling in dropdown selection\"\n```\n\n## 🔄 Pull Request (PR)\n\n### 1. Actualizar tu Rama\n\n```bash\n# Obtener cambios de upstream\ngit fetch upstream\n\n# Hacer rebase si es necesario\ngit rebase upstream/main\n# O merge (menos preferible)\ngit merge upstream/main\n```\n\n### 2. Push a tu Fork\n\n```bash\ngit push origin feature/tu-feature\n```\n\n### 3. Crear PR en GitHub\n\n**Título:** Descriptivo y corto\n```\nfeat: Add new test suite for user registration\nfix: Correct element timeout issue in checkout\n```\n\n**Descripción PR:**\n\n```markdown\n## Descripción\nBreve descripción de los cambios.\n\n## Tipo de Cambio\n- [ ] Nuevo test\n- [ ] Page Object nuevo\n- [ ] Corrección de bug\n- [ ] Documentación\n- [ ] Refactorización\n\n## Cómo Verificar\n- [ ] npm run test\n- [ ] npm run test:headed\n- [ ] npm run test:ui\n\n## Checklist\n- [ ] Tests pasan localmente\n- [ ] Código sigue convenciones del proyecto\n- [ ] JSDoc completo en métodos nuevos\n- [ ] Sin código comentado\n- [ ] README actualizado si es necesario\n- [ ] No hay conflictos con main\n\n## Referencias\nCloses #123 (si aplica)\n```\n\n### 4. Review Process\n\n- Espera feedback de los mantainers\n- Actualiza según sugerencias\n- Resuelve conflictos si existen\n- Una vez aprobado, será mergeado\n\n## 🧪 Testing Guidelines\n\n### Principios de Tests\n\n1. **Independencia:** Cada test debe funcionar solo\n2. **Claridad:** Nombres y código autosexplicativos\n3. **Aislamiento:** No compartir estado entre tests\n4. **Determinismo:** Resultados consistentes\n5. **Velocidad:** Optimizar para ejecución rápida\n\n### Estructura de Test\n\n```typescript\ntest('should describe what happens', async ({ page }) => {\n  // Arrange: Preparar datos y estado\n  const testData = { /* datos */ };\n  \n  // Act: Realizar la acción\n  const result = await somePage.doSomething(testData);\n  \n  // Assert: Verificar resultado\n  expect(result).toBe(expectedValue);\n});\n```\n\n### Evitar Anti-patterns\n\n```typescript\n// ❌ NO HACER\nawait page.waitForTimeout(5000);\n\n// ✅ HACER\nawait page.waitForLoadState('load');\nawait element.waitFor({ state: 'visible' });\n```\n\n## 📚 Recursos\n\n- [Playwright Documentation](https://playwright.dev)\n- [Page Object Model Pattern](https://docs.microsoft.com/en-us/archive/msdn-magazine/2012/december/msdn-magazine-automation-patterns-page-object-pattern)\n- [TypeScript Best Practices](https://www.typescriptlang.org/docs/handbook/declaration-files/do-s-and-don-ts.html)\n- [Jest Testing Best Practices](https://kentcdodds.com/blog/common-mistakes-with-react-testing-library)\n\n## 🆘 Ayuda\n\n- 📧 Para dudas: Abre una Discussion\n- 🐛 Para bugs: Abre un Issue\n- 💡 Para sugerencias: Abre un Issue de Feature Request\n\n## 📄 Licencia\n\nAl contribuir, aceptas que tu código esté bajo la licencia del proyecto (MIT).\n\n---\n\n**¡Gracias por contribuir a QAutomation01!** 🎉"
